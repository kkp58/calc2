A design pattern provides a general reusable solution for the common problems that occur in software design. The pattern typically shows relationships and interactions between classes or objects. By using design patterns, one can make code more flexible, reusable, and maintainable. There are mainly three types of design patterns - Creational, Structural and Behavioral

  1. Creational design patterns are the Factory Method, Abstract Factory, Builder, Singleton, Object Pool, and Prototype. It involves how you create objects.

    In many real-world situations, we want to create only one instance of a class. For example, in our calculator program, we have only one 'history'(static property) which stores all the calculations done before. This pattern is called a Singleton pattern.

    Suppose we want to create multiple instances of a similar kind and want to achieve loose coupling then we go for Factory pattern. A class implementing a factory design pattern works as a bridge between multiple classes. For our calculator program, we have used factory to create objects like different calculations(addition, subtraction, etc). Through loose coupling it allows us to add new types of products without disturbing previous code.

  2. Structural design patterns are Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Private Class Data, and Proxy. It is how you organize the program

    Facade - Any library does many things for you, but we need only a few things for our program. So you put a facade(a structure) around the library to only do the specific things you need and even while testing, it only tests for the functionalities you have used. So facade wraps up the functionality of any library and then provides easy user access. When you wrap the commands, that is called facade. It is an overall class.
    For our example, the ‘Calculator’ class wraps the ‘calculations’ class in a facade and provides the interface to send the commands to the ‘calculations’ class.

  3. Behavioral patterns are Chain of responsibility, Command, Interpreter, Iterator, Mediator, Memento, Null Object, Observer, State, Strategy, Template method, Visitor. It is like how your program reacts to things.

    Command - A behavioral design pattern that converts requests or simple operations into objects. One line of code. Commands are sent to the calculations library to do what they need to do.
    For example, in our calculator program, we are giving set of methods to provide commands like add, subtract, multiply, divide, add calculation, get_results and so on. Our code is the set of commands that perform different operations.
